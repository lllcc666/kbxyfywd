# 任务自动化全量目录与分析流程

本文档把 `data/taskinfo.xml` 的全部根任务一次性读出来，但分析顺序仍然按“根系列 -> 子任务链 -> 运行时节点 -> packet recipe -> hook”逐层推进。

## 1. 目标

- 先确认每个根任务系列的边界，再确认它的执行顺序。
- 先读 XML 骨架，再读 AS3 解释，最后才落到自动化。
- 不把 UI 主题名当执行顺序，不把任务名当主键。

## 2. 代码优先级

1. `data/taskinfo.xml`
2. `TaskInfoXMLParser.as`
3. `TaskXMLParser.as`
4. `TaskList.as`
5. `TaskView.as` / `TaskControl.as`
6. `src/hook/wpe_hook.cpp`

## 3. 标准分析流程

1. 定位根任务。先拿 `taskID`，再看它属于哪个 family。
2. 看骨架。统计子任务数，先分单步、短链、中链、长链。
3. 看约束。读取 `after` / `before`，确认是否受前置任务限制。
4. 看运行时解释。用 `TaskInfoXMLParser` 和 `TaskList` 还原可见性、接取条件、完成状态。
5. 看节点类型。用 `TaskXMLParser` 分出 `alert`、`desc`、`choose`、`battle`、`flash`、`ai`、`otherpopup`、`condition`、`targetScene`，再把它们翻译成具体发包动作。
6. 还原执行顺序。XML 顺序只做初始参考，运行时顺序、抓包顺序和当前 hook 的固定 recipe 优先。
7. 再写自动化。先写可复用约束，再补特殊分支和每一步的回包点。

## 4. 约束与判定

- 不能一次性把 350 个根系列都写成完整自动化。
- 不能只看任务名，必须同时看 `taskID`、`subtaskID`、`after`、`before`、`type`、`diff`。
- `tasktheme.xml` 只影响 UI 分组，不代表真实执行顺序。
- 根任务名可能重复，必须以 ID 做主键。
- XML 顺序和运行时顺序冲突时，以运行时为准。
- 只要出现 `battle`、`flash`、场景切换或特殊收口，就不能套用普通对话模板。
- `choose flag=3` 一类选择分支不能当普通 `desc` 处理。
- `TaskList` 的状态判断只看首尾子任务，不能只看名字。
- 每一条记录都必须写清“来源 -> 发送 -> 回包 -> 恢复点”，只列名称不列包序列不算完成。

## 5. XML / AS3 规则

### 5.1 `taskinfo.xml`

- 根节点是 `task`。
- 根任务只定义骨架，不定义最终执行器。
- `after` / `before` 是前置/后置约束，不是简单任务号。
- 子节点才是自动化的实际拆分单位。

### 5.2 `TaskInfoXMLParser`

- `parse()` 会把每个根任务展开成扁平的 `Task` 列表。
- `parseClassfy()` 会先过滤 `off`，再按 family 过滤，再做前置/后置可见性判断。
- family 1 和 7 的已完成任务会被单独隐藏，不要按静态 XML 直接推导列表。
- `canInputOrNot()` 才是前置判断核心。
- `getPrefixListByAfterID()` 说明 `after` 不是纯任务号，而是映射锚点。
- `parseTargetScene()`、`parseAward*()`、`getGoodsInTaskInfo()` 是具体落地字段。

### 5.3 `TaskList`

- 状态分三类：未接、已接、已完成。
- `getStateOfSpecifiedTask()` 只看首尾子任务，不看中间描述。
- `getSubtaskList()` 先按状态拼数组，再按 `subtaskID` 排序。
- `updateTask()`、`complete()`、`drop()` 决定状态流转。

### 5.4 `TaskXMLParser`

- `alert`：弹出提示，不算普通任务节点。
- `desc`：普通描述节点，可内嵌 `choose`。
- `choose`：分支节点，必须按选择值拆分。
- `battle`：战斗节点，先起战斗，再处理战后收口。
- `flash`：过场/特效节点，可能受性别和战后条件影响。
- `otherpopup`：独立弹层，不要并入普通对话。
- `condition`：完成条件，不是纯展示文本。
- `targetScene`：直接场景跳转。

### 5.5 对话与选择

- 选择分支不是只发一次任务对话包。
- 运行时会先发训练信息，再发任务对话。
- 这类顺序必须保留，不要合并成单包。

### 5.6 UI 只是展示层

- `TaskArchivesVersion3` 和 `SortedTaskList` 只负责树状展示。
- `TreeRootNode` 里硬编码的标题只是 UI 文案，不是执行链名称。
- 真正的自动化依据还是 `taskinfo.xml` + AS3 + hook。

## 8. 后续批次顺序

1. 先补已验证或短链系列。
2. 再补 family 1、4、6、7、9 这种根数少的系列。
3. 再补 family 2 的主线章节。
4. 最后补 family 3 的超大支线池，并按 1000 段继续拆。
5. 每一批都必须保留 `已验证` 和 `待验证` 两个字段。

## 10. 统一模板

- 根任务：`taskID / 名称 / type / diff / 子任务数`
- 结构：`单步 / 短链 / 中链 / 长链`
- 证据：`已验证 / 已采集 / 待验证`
- 执行顺序：`XML 顺序 / 运行时顺序 / hook 顺序`
- packet：`alert / desc / choose / battle / flash / ai / otherpopup / condition`
- 中断与恢复：`失败点 / 回收点 / 重试点 / 跳步点`

## 11. 结论
